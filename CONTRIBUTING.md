# 기여 가이드

## 브랜치 전략

```
main          ← 안정 버전 (멘토 승인 후 머지)
├── feature/  ← 기능 개발
├── docs/     ← 문서 작업
└── fix/      ← 버그 수정
```

**브랜치 네이밍**: `feature/이슈번호-간단설명`
- 예: `feature/3-mmlu-survey`, `docs/5-evaluation-criteria`

---

## 작업 흐름

1. GitHub Issues에서 작업할 이슈 확인 (또는 새 이슈 생성)
2. 이슈에 자신을 assign
3. 브랜치 생성: `git checkout -b feature/이슈번호-설명`
4. 작업 수행 및 커밋
5. Push 후 Pull Request 생성
6. 멘토 리뷰 후 머지

---

## 커밋 메시지 규칙

```
[타입] 간단한 설명

본문 (선택)
```

타입:
- `[조사]` — 벤치마크 조사/분석
- `[설계]` — 평가 기준/문제 설계
- `[구현]` — 코드 구현
- `[문서]` — 문서 작성/수정
- `[수정]` — 버그 수정

예시:
```
[조사] MMLU 벤치마크 분석 완료
[구현] 벤치마크 실행 스크립트 초안 작성
[문서] 프로젝트 계획서 업데이트
```

---

## 문제 제작 가이드

벤치마크 문제는 `benchmarks/categories/` 에 JSON 형식으로 저장합니다.

파일명: `{카테고리}_{난이도}_{번호}.json`
- 예: `coding_l2_001.json`, `stem_l3_005.json`

필수 필드:
- `id`: 고유 ID (카테고리약자-L레벨-번호)
- `category`: 카테고리명
- `level`: 난이도 (1-4)
- `title`: 문제 제목
- `prompt`: AI에게 줄 프롬프트
- `scoring`: 채점 기준

---

## 코드 스타일

- Python: PEP 8 준수
- 문서: 한국어 작성 (코드 주석 포함)
- JSON: 2칸 들여쓰기, UTF-8
