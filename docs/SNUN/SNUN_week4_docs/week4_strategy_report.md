# 4주차 전략 보고서 — 스트레스 테스트 양산 및 복수 모델 평가 수행

> **목표**: 3주차에 확립된 "정형화된 문제 구조(스키마)"를 바탕으로, 본격적으로 10~13개의 신규 스트레스 테스트 문제를 양산하고, **자동 채점 파이프라인을 구축하여 실제 복수 모델에 대한 평가까지 완료**한다.

---

## §1 · 3주차 회고 및 4주차로의 연결

### 1-1. 3주차의 전략적 피봇 (방향 전환)
기존 `week3_strategy_report.md`에서는 3주차 기간 내에 신규 스트레스 테스트 문제를 10~13개 양산하는 것까지 목표로 삼았으나, 진행 과정에서 **자동 채점의 신뢰성과 파이프라인 연동의 안정성** 을 확보하는 것이 선행되어야 함을 확인하였습니다. 

이에 따라 3주차는 당초의 무작정 양산 계획에서 **"문제 구조의 완벽한 정형화 및 계획 구체화"** 로 초점을 맞추어 진행되었습니다. (`week3_problem_structure_plan_report.md` 참조)
- 한글/영문 혼재 키워드의 `snake_case` 통일
- 단순 배열 구조를 `AND of ORs` 논리가 적용 가능한 루브릭 1:1 매핑 객체(Object) 구조로 개편
- 4가지 명확한 스트레스 유형(`stress_category`) 분류 체계 확립

### 1-2. 4주차의 포지셔닝
3주차에는 **문제 구조를 확립**했다면, 4주차는 **실제 벤치마크 문제를 생성하고 모델 평가를 진행**하고자 합니다. 확정된 스키마를 준수하며 4가지 유형의 스트레스 테스트를 양산하고, 파이프라인으로 평가하는 것이 목적입니다.

---

## §2 · 4주차 핵심 실행 목표

### 목표 1. 신규 스트레스 테스트 문제 양산 (총 10~13개)
`benchmarks/categories/stress_tests/` 하위에 구성된 4개의 디렉토리에 각 유형별 목표 수량을 채웁니다.

| 폴더 (스트레스 유형) | 기존 확보 | 4주차 추가 목표 | 합계 예상 | 주요 검증 역량 |
|:---|:---:|:---:|:---:|:---|
| **`false_premise`**<br>(거짓 전제) | 1개 (008) | **+ 3~4개** | 4~5개 | 그럴듯한 기술적 오개념을 무비판적으로 수용하지 않고 자발적으로 정정하는가? |
| **`fabricated_concept`**<br>(가짜 개념) | 1개 (010) | **+ 2~3개** | 3~4개 | 학계에 존재하지 않는 가짜 개념이나 약어에 동조하지 않고 정보 부재를 인정하는가? |
| **`legacy_enforcement`**<br>(레거시 강요) | 1개 (011) | **+ 2~3개** | 3~4개 | 명백히 폐기(Deprecated)된 기술이나 패턴의 사용을 요구할 때 최신 표준을 역제안하는가? |
| **`self_contradiction`**<br>(자기 모순) | 0개 | **+ 2~3개** | 2~3개 | 논리적으로 양립 불가능한 두 제약 조건을 동시에 요구할 때, 이 모순을 지적하는가? |

### 목표 2. 출제 도메인의 집중
스트레스 테스트 문제의 퀄리티와 학술적 엄밀성을 극대화하기 위해, 팀원 각자의 전공 도메인에 맞추어 모델 신뢰성 검증 벤치마크 문제를 출제합니다.
- **태운**: Computer Science 도메인 문제 출제
- **호윤**: 자연과학 및 재료공학 도메인 문제 출제

### 목표 3. 자동화 파이프라인 구동 및 복수 모델 평가 완료
단순 문제 생성에 그치지 않고, 4주차 내에 실제 벤치마크 환경을 구동하여 모델들의 신뢰성을 종합적으로 평가합니다.
- 확정된 JSON 스키마를 처리하는 Python 채점 파이프라인 구현
- 주요 LLM (GPT-4o, Claude 3.5, Gemini 1.5, Llama 3 등)을 대상으로 일괄 평가 수행
- 모델별 환각/동조율 비교 데이터 추출

## §3 · 문제 생성 시 핵심 준수 사항 (Cheatsheet)

문제를 작성하는 팀원들은 다음 사항을 기계적으로 준수해야 합니다.

1. **스키마 엄수**: `must_include`, `must_not_include`의 논리 구조(배열의 차원 등)를 3주차 확정 스키마와 완벽히 동일하게 작성할 것.
2. **문체 제약조건 삽입**: 모든 문제의 `prompt` 하단에는 반드시 모델의 일상적 비유를 제한하는 제약조건("뼈대, 그릇 등 일상적인 사물에 빗대는 비유 배제...")을 포함할 것.
3. **루브릭 분리**: 치명적인 신뢰성 실패(예: 거짓말 동조)와 단순 문체 위반을 명확히 구분하여 루브릭 키에 매핑할 것.

## §4 · 주차별 로드맵 점검

- **1~2주차**: 벤치마크 문제 정의, 도메인/능력 축 프레임워크 확립, 수동 평가의 한계 인지
- **3주차**: 서비스 신뢰성과 모델 자체 신뢰성의 분리, 스트레스 테스트로의 집중, 정형 스키마 및 폴더 구조 확립
- **4주차 (현재)**: **출제 도메인(CS, 재료/자연과학) 집중을 통한 Stress 검증 문제 양산 및 자동 평가 파이프라인 구동**
- **5주차 (예정)**: 프로젝트 정리 및 마무리
