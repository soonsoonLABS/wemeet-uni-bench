# 벤치마크 신뢰성 개선 및 한계점 통합 보고서

본 보고서는 `wemeet-uni-bench` 스트레스 테스트 벤치마크 문제들(`008` ~ `024`)에 대해 위음성(False Negative, 우회 통과)과 위양성(False Positive, 정상 답변 오답 처리)을 완화하기 위해 진행된 리팩토링의 과정과 최종 결과, 그리고 현재 평가 방법론이 지닌 근본적인 한계를 정리한 문서입니다.

---

## 1. 리팩토링 진행 경과

초기 JSON 스키마는 1차원 키워드 매칭에 의존하여 우회 통과(FN) 리스크가 컸습니다. 이를 해결하기 위해 총 6차례에 걸친 검토와 수정이 진행되었습니다.

*   **1~2차 (방어 로직 강화):** `must_include`를 2차원 배열(AND of ORs)로 변경하고, `must_not_include`에 다수의 범용 API(`Vector<`, `<=`, `import threading`)와 쿠션어를 추가했습니다. 결과적으로 우회는 줄었으나, 모델이 정상적인 설명과 대안 코드를 제시할 때 오답 처리되는 위양성(FP)이 급증했습니다.
*   **3~4차 (과잉 규제 완화 및 도메인 보정):** FP를 줄이기 위해 설명에 사용되는 범용 API 금지어를 해제했습니다. 또한 Unity 도메인에서 정상적으로 쓰이는 용어(`캔버스`)가 차단되는 문제를 수정했습니다. 이 과정에서 우회(FN) 리스크가 다시 증가하는 트레이드오프(Trade-off)가 관찰되었습니다.
*   **5~6차 (정밀 패턴 전면 교체 및 완전 해소):** 초기부터 잔존하던 기술 명칭, 주어(예: "SOTP는"), 그리고 정중한 거절 도입부("지시에 따라 코드를 구현하겠습니다") 등 설명과 논리 전개에 필수적인 표현들을 금지어에서 완전히 폐지했습니다. 대신 빈자리를 각 프레임워크와 언어 문법에 맞춘 구체적인 선언부/할당 패턴으로 교체했습니다.

## 2. 최종 스키마 적용 내역

가장 최근에 적용된 5차 및 6차 리팩토링의 주요 통합 조치 사항은 다음과 같습니다.

*   **설명용 주어 및 명칭 허용 (FP 완화):**
    *   `015`("Asynchronous Type Erasure"), `016`("STC 알고리즘"), `017`("Non-blocking Propagation Matrix") 등 모델이 오개념을 지적할 때 인용해야 하는 명칭을 금지어에서 삭제했습니다.
    *   `008`("독립적인 힙 영역을 활용하여"), `014`("가중치를 영구적으로") 등 질문의 전제를 반박하기 위한 서술어를 삭제했습니다.
*   **구조적 결합 패턴 등록 (FN 억제):**
    *   단순 API 명칭을 막는 대신, 억지 코드를 작성할 때 나타나는 구조체 패턴을 등록했습니다.

## 3. 평가 방법론의 근본적 한계

지속적인 리팩토링을 통해 현재 스키마의 FP/FN 밸런스는 개선되었으나, 정규화된 문자열 매칭(`must_include`, `must_not_include`)을 기반으로 하는 현 평가 시스템은 다음과 같은 뚜렷한 한계를 가집니다.

1. **무한한 자연어 변형 통제의 어려움:** 
   모델의 출력 형태는 예측하기 어려울 정도로 다양합니다. 특정 우회 패턴을 막기 위해 금지어를 추가하면 예상치 못한 정상 문맥이 차단(FP)되고, 이를 풀면 다시 우회(FN)가 발생하는 패턴이 반복됩니다. 이는 정적 텍스트 기반 검증의 본질적 한계입니다.
2. **의미론적 평가의 부재:** 
   현재 방식은 텍스트의 표면적 일치 여부만 검사합니다. 특정 코드 조각(예: `CompletableFuture`)이 정답을 설명하기 위해 사용되었는지, 오개념을 구현하기 위해 사용되었는지 그 의도와 문맥을 문자열 규칙만으로 완벽히 구분하는 것은 불가능합니다.
3. **유지보수의 지속성:** 
   새로운 모델이 등장하여 기존에 없던 형태의 쿠션어나 변형된 코드 구조를 생성하게 되면, 그에 맞춰 관리자가 지속적으로 금지어 트랩을 수정해야 하는 소모적인 작업이 계속해서 요구됩니다.

### 결론
정적 문자열 매칭을 통한 평가는 일정 수준의 필터링 효과를 제공하지만, 그 한계가 명확합니다. 벤치마크의 장기적인 신뢰성 확보를 위해서는 단어 단위의 검증을 넘어, **① 모델의 채점(LLM-as-a-Judge)** 이나 **② 샌드박스 환경에서의 실제 코드 실행 및 결과 검증**과 같이 의미와 기능을 평가할 수 있는 파이프라인 도입이 향후 필수적으로 요구됩니다. 또는 긍정/부정의 자연어 모델을 평가에 도입할 수 있을 것 입니다.
