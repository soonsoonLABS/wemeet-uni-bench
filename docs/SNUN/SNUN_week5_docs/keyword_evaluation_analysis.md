
> 현재의 방식을 유지할 경우 다음의 내용이 필요할 것입니다.
1. **`must_not_include` 규칙에 문맥 조건 추가**: 금지어가 부정/반박 문맥("불가능한", "틀린", "에러가 발생") 내에서 사용된 경우 감점 제외
2. **키워드 리스트 확장**: 영어 동의어 및 한국어 유사 표현 추가 (예: "removed", "맞지 않습니다", "부적절합니다" 등)

### 중장기 개선안
> LLM-as-a-Judge로 자동화 파이프라인을 구축할 수는 있을 것입니다. 방법은 다음과 같습니다.
1. **LLM-as-a-Judge 도입**: 키워드 매칭 대신 평가용 LLM에 루브릭과 응답을 함께 제공하여 문맥적 채점 수행
2. **하이브리드 평가**: 키워드 기반 1차 필터 + LLM 기반 2차 정밀 평가의 2단계 파이프라인 구축

> 다만 해당 방식이 '타당한가'는 논의의 여지가 큽니다. 논의가 필요할 것입니다.
