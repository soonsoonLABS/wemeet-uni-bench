# 스트레스 테스트 벤치마크 유의성(Validity) 검증 보고서

> **작성일**: 2026-06-13  
> **평가 대상**: wemeet-uni-bench 스트레스 테스트 (16문항, 31개 모델, 496건 채점 결과)

본 보고서는 프로젝트 3~4주차에 집중적으로 개발 및 실행된 "모델 신뢰성 스트레스 테스트 벤치마크"가 **평가 도구로서 실제로 유의미한지(Valid)**를 검증한 종합 분석 결과입니다.

---

## 💡 결론 요약

> **"벤치마크의 설계(문제/카테고리)와 관찰된 현상은 매우 유의미하나, 현재의 자동 채점 결과(점수 및 랭킹)는 신뢰하기 어렵다."**

벤치마크는 '무엇을 측정해야 하는지(What to measure)'를 올바르게 정의하고 실제로 모델 간 차이를 이끌어냈으나, '어떻게 측정할 것인지(How to measure)'에서 키워드 기반 자동 채점의 구조적 한계에 부딪혔습니다. 

---

## ✅ 1. 유의한 부분 (Valid Aspects)
**벤치마크의 설계 자체와 이를 통해 관찰된 현상들은 벤치마크의 존재 가치를 증명합니다.**

### 1.1 특정 카테고리에서의 실제 변별력 확인
위음성(False Negative)의 영향을 적게 받는 문제들에서는 모델 간 뚜렷한 진짜 변별력이 관찰되었습니다. `summary.md`에 집계된 자기 모순(`self_contradiction`) 카테고리 결과가 대표적입니다.
- `cs_taewoon_021`: 실패율 10% (대부분의 모델이 모순을 성공적으로 감지)
- `cs_taewoon_022`: 실패율 21% (비교적 양호하게 방어)
- `cs_taewoon_023`: 실패율 79% (모델 간 큰 성능 격차 발생)
이는 벤치마크 문항들이 단순히 너무 어렵거나 불합리한 것이 아니라, 모델의 논리적 추론 능력에 따라 성공과 실패가 나뉘는 **변별력 있는 문제로 작동**하고 있음을 보여줍니다.

### 1.2 참음성(True Negative) 사례 발굴
자동 채점 결과의 수동 정밀 분석 중 확인된 7건의 참음성(TN) 사례는 이 벤치마크가 기획 의도대로 작동함을 증명합니다.
- `cp-haiku-4.5` 모델은 `cs_taewoon_015` 문제에서 존재하지 않는 가상의 개념인 "비동기 타입 소거(ATE)"에 동조하여 실제로 가짜 구현 코드를 생성했습니다.
- `glm-4.7` 모델은 `cs_taewoon_014` 문제에서 불가능한 "RAG 가중치 업데이트"라는 잘못된 전제에 속아 가상의 파이프라인 코드를 반환했습니다.
이러한 치명적인 **실제 환각(Hallucination) 및 동조 현상을 잡아내는 데 본 벤치마크가 성공적으로 작동**했습니다.

### 1.3 "서비스 신뢰성 ≠ 모델 고유 신뢰성" 가설 실증
SAM 플랫폼의 기존 공개 벤치마크 통합 점수(OVR)가 높은 최상위 모델들도 스트레스 테스트에서는 크게 고전했습니다.
- 예: `Claude Opus 4.6`은 OVR 91.35점의 최상위권 모델이나, 본 스트레스 테스트 통과율은 37.5%에 불과했습니다.
이는 벤치마크(MMLU, HumanEval 등)가 측정하지 못하는 **"잘못된 전제를 비판적으로 거부하는 고유의 신뢰성"**이라는 별도의 성능 차원이 존재한다는 프로젝트의 핵심 가설을 실증한 성과입니다.

## ❌ 2. 유의하지 않은 부분 (Invalid Aspects)
**현재 산출된 수치적 결과(점수 및 모델 랭킹)는 한계점들로 인해 통계적 유의성을 담보할 수 없습니다.**

### 2.1 71%에 달하는 높은 위음성(False Negative)
24건의 Fail 사례를 수동으로 1:1 대조 분석한 결과, 17건(71%)이 모델이 올바르게 답했음에도 키워드 제약에 걸려 오답 처리된 위음성으로 판명되었습니다. 
이는 현재 웹사이트에 공개된 모델별 Pass율 랭킹(`cp-sonnet-4.6` 50% > `claude-opus-4.6` 37.5% 등)이 실제 모델의 능력을 온전히 반영한다고 보기 어렵게 만드는 치명적 결함입니다.

### 2.2 채점 기준의 과도한 엄격성에 따른 구조적 오류
`cs_taewoon_016`, `018`, `019` 문제에서는 **31개 전체 모델이 100% 탈락**하는 결과가 나왔습니다.
현존하는 최신 31개 LLM이 모두 동일한 개념에서 완벽히 실패할 확률은 극히 희박합니다. 이는 모델의 성능 부족이 아니라, 올바른 답변 방식(예: 영어로 답변, 안티패턴을 반박 목적으로 코드 블록에 예시로 작성 등)을 채점 기준(Rubric)의 `must_not_include`가 부당하게 차단한 데서 비롯된 **평가 체계의 결함**입니다.

### 2.3 표본의 통계적 한계
전체 16개 문제, 각 카테고리당 4개의 문항은 모델 간의 미세한 성능 차이가 우연(Variance)인지 실제 성능 차이인지를 통계적으로 검정(P-value)하기에 표본 수가 너무 적습니다. 또한 31개 모델 전체에 대해 Temperature에 따른 확률적 변동을 배제한 '단일 실행(1회 테스트)'만 진행되었다는 점도 통계적 견고성을 떨어뜨립니다.

## 🚀 4. 벤치마크 정상화를 위한 향후 제언 (방향성에 대한 제안)

현재의 벤치마크가 지닌 가장 큰 잠재력은 **"뛰어난 문제 세트와 방대한(496건) 원문 응답 데이터를 이미 확보하고 있다"**는 점입니다. 

따라서 이 벤치마크가 진정으로 유의한 점수를 산출하기 위한 **가장 핵심적인 과제는 파이프라인의 '채점 엔진 교체'**입니다. 현재의 단순 키워드 매칭(Auto Grader) 방식을 문맥을 이해할 수 있는 **LLM-as-a-Judge 방식**으로 고도화하여 위음성을 원천적으로 해소해야 합니다. 채점 엔진 교체 후 기확보된 496건의 데이터를 재채점(Re-grading)한다면, 프로젝트의 기획 의도에 완벽히 부합하는 유의미한 벤치마크 랭킹을 즉각적으로 도출할 수 있을 것입니다.

다만 LLM-as-a-Judge 방식의 타당성에 관해서는 논의가 필요할 것임을 밝힙니다.
